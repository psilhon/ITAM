## 场景分析

### ✅ 场景 1：健康检查

- **表现优秀**。RPS 达到 742，p99 延迟 229ms
- 包含一次 DB 查询（`SELECT 1`），单进程处理 700+ RPS 完全够用
- 对应实际使用中健康检查探针压力，毫无问题

### ✅ 场景 2：服务器列表查询

- **表现良好**。RPS 244，p95 延迟 380ms
- 包含分页、搜索、排序等复合查询
- p99 稳定在 400ms 以内，用户体验良好

### ✅ 场景 3：Dashboard 聚合统计

- **表现良好**。RPS 333，p95 延迟 277ms
- 该接口执行 10+ 次并行 DB 查询（count + groupBy + findMany）
- 并行查询策略有效，多表聚合耗时可控

### ✅ 场景 4：CRUD 生命周期

- **表现优秀**。RPS 120（串行），单次操作平均仅 8ms
- Create → Read → Update → Delete 全流程零失败
- 串行场景下（模拟真实用户操作）完全胜任

### ⚠️ 场景 5：并发写入压力

- **发现问题**。20 并发写入时，90% 请求超时失败
- **错误**：`PrismaClientKnownRequestError: Operations timed out`
- **根因**：SQLite 文件数据库的 **写入锁竞争**

```
SQLite 写入模型：
┌─────────┐     ┌──────────┐
│ 请求 A  │────→│ SQLite   │ ← 独占写锁（一次只允许一个写入）
│ 请求 B  │────→│ 文件锁   │ → 等待... 超时！
│ 请求 C  │────→│ 等待队列  │ → 等待... 超时！
└─────────┘     └──────────┘
```

- 平均延迟 9 秒（10 秒超时阈值），说明写入严重排队
- **影响范围**：仅影响高并发写入场景（>5 并发写），正常使用不受影响

### ✅ 场景 6：限流测试

- 临时调高到 10000 次后，350 次请求全部通过
- 默认 300 次/15分钟 的限流在正常使用中不会触发（单人操作频率远低于此）

## 性能评估

### 优势

| 维度 | 评估 | 说明 |
|------|------|------|
| 读性能 | ⭐⭐⭐⭐⭐ | 700+ RPS 健康检查，250+ RPS 列表查询，完全满足单用户/小团队使用 |
| 查询效率 | ⭐⭐⭐⭐ | Dashboard 多表聚合 p95 < 300ms |
| 单次 CRUD | ⭐⭐⭐⭐⭐ | 平均 8ms，几乎无感知延迟 |
| 限流保护 | ⭐⭐⭐⭐⭐ | 300 次/15分钟，有效防暴力攻击 |

### 瓶颈

| 维度 | 评估 | 说明 |
|------|------|------|
| 并发写入 | ⭐⭐ | SQLite 独占写锁，>5 并发写入即开始超时 |
| 高并发扩展 | ⭐⭐ | 单进程 Node.js + 单文件 SQLite，无法水平扩展 |

## 优化建议

### 🔧 短期优化（无需架构变更）

1. **增加 SQLite 连接超时**
   - 在 `DATABASE_URL` 中添加超时参数：`file:dev.db?connection_limit=1&pool_timeout=30`
   - 或在 Prisma schema 中配置

2. **写入队列化**
   - 后端增加内存队列，将并发写入串行化
   - 避免 SQLite 锁竞争，牺牲吞吐量换取成功率

3. **Prisma 事务超时**
   ```prisma
   datasource db {
     provider = "sqlite"
     url      = env("DATABASE_URL")
     timeout  = 30 // 增加到 30 秒
   }
   ```

### 🏗️ 长期优化（架构升级）

如果未来需要支持多用户并发写入：

| 方案 | 复杂度 | 效果 | 适用场景 |
|------|--------|------|---------|
| 迁移到 PostgreSQL | 中 | 彻底解决并发问题 | 多用户 >10 人 |
| 使用 better-sqlite3 | 低 | 比当前 Prisma SQLite 快 3-5x | 保持 SQLite 但需要更高性能 |
| 添加 Redis 缓存 | 中 | 读性能翻倍 | 读多写少场景 |
| PM2 cluster 模式 | 低 | 利用多核 CPU | 需配合连接池数据库 |

## 结论

ITAM 系统在**读操作和低并发写入场景下表现优秀**，完全满足个人或小团队（<10 人）使用需求。唯一瓶颈是 SQLite 的并发写入限制，但对于资产管理系统的实际使用场景（以读和低频写入为主），这不会成为问题。

**综合评分：4/5** ⭐⭐⭐⭐
